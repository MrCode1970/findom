# AGENTS.md

Инструкции для Codex-агентов, работающих в репозитории `findom`.

---

# 1. Цель репозитория

Репозиторий `findom` реализует систему автоматического учета личных финансов.

Основная задача системы:

источники данных → коннекторы → нормализация → витрина данных → Google Sheets

Основные цели проекта:

* автоматический сбор транзакций из источников
* нормализация движений
* хранение чистого журнала операций
* синхронизация с Google Sheets витриной

Текущий основной provider:

cal_digital

# 3. Рабочие принципы

Агент должен соблюдать следующие принципы:

* Делать минимальные изменения, необходимые для выполнения задачи
* Не менять нерелевантные файлы
* Не изменять артефакты в `.state/`, если это не требуется задачей
* Не рефакторить код без явного запроса
* Избегать масштабных изменений архитектуры

При поиске по коду использовать:

rg
rg --files

# 6. Архитектура коннекторов

Каждый provider должен следовать архитектуре:

fetch → api → converters

Назначение слоев:

fetch

* сетевые запросы
* получение сырых данных

api

* логика взаимодействия с API
* pagination
* retry

converters

* преобразование в единый формат транзакций

---

# 7. Контракт транзакций

Нормализованная транзакция должна содержать:

external_id
date
amount
currency
account
category
description

Ключевые требования:

* external_id обязан быть уникальным
* дата должна быть нормализована
* сумма должна иметь корректный знак
* refund должен обрабатываться корректно

Нельзя добавлять поля, которые нельзя воспроизвести из исходных данных.

---

# 8. Правила витрины данных (Google Sheets)

Google Sheets используется как витрина данных, а не как источник истины.

Источник истины:

нормализованные транзакции

Правила:

* витрина должна быть полностью воспроизводима
* нельзя создавать поля, требующие ручного заполнения
* категоризация должна использовать справочник категорий
* свободный текст категорий запрещён

Типовая структура витрины:

Transactions (сырые)
Normalized (нормализованные)
Categories (справочник)
Dashboard (сводка)

pip install -e .
python -m playwright install chromium

---

# 10. Полезные команды

Основная команда:

python -m findom

Синк транзакций:

python tools/connectors/run_sync.py --provider cal_digital

Dry run:

python scripts/run_sync.py --source cal_digital --dry-run

Тесты:

pytest -q

---

# 11. Правила изменения кода

Требования:

* Python >= 3.10
* новые provider должны соблюдать архитектуру fetch → api → converters
* изменения нормализации требуют тестов
* изменения схем требуют сверки с

tables/schema/*

---

# 12. Контракт результата задачи

Финальный отчёт агента должен содержать:

1. Что изменено
2. Какие файлы изменены
3. Какие команды запускались
4. Результат проверок
5. Возможные риски

---

# 13. Правила задач по дизайну таблиц

Если задача связана с дизайном витрины данных:

Каждый вариант должен создавать:

docs/sheets_design/<variant>.md
docs/sheets_design/<variant>.json

JSON-файл должен описывать:

* листы
* таблицы
* диапазоны
* формулы
* правила форматирования
* демо-данные

Все варианты должны быть сопоставимы.

---

# 14. Критерии оценки дизайна витрины

При генерации нескольких вариантов необходимо оценивать:

1. Наглядность на одном экране
2. Скорость ручного ввода
3. Масштабируемость (1000+ транзакций)
4. Риск ошибок
5. Совместимость с автоматическим импортом

---

# 15. Приоритеты разработки

При принятии решений соблюдать принцип:

Stability over Speed

Предпочтение отдаётся:

* стабильности
* воспроизводимости данных
* прозрачности логики
